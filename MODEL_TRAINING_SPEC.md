## TARGET 1 — Reranker (CrossEncoder) · PRIMARY

**Why first:** 151 reranker_failures vs 43 retrieval_failures. 111 queries are
"near-miss" — the best expected file sits at pool rank 11-50, i.e. the reranker
is one good decision away. The pool is now clean (FIX-A removed doc-noise,
FIX-D removed duplicate chunks) so the reranker will train on a realistic
candidate distribution, not noise.

- **Current model:** `Tarshevskiy/pay-com-rerank-l12-ft-run1`.
- **Training data:** `bench_runs/training_data/reranker_pairs.jsonl` — regenerated
  from the post-fix pipeline:
  - **3054 positive** pairs `(query, expected_file, label=1)`
  - **4520 hard-negative** pairs `(query, wrong_file, label=0)` — each hard
    negative is a file the *current* reranker puts in the top-10 ABOVE an
    expected file. This is the exact signal to correct.
  - each row has `content_excerpt` (first chunk) for direct CrossEncoder input.
- **Split:** query-disjoint train/test (do NOT let the same query_id appear in
  both — past runs leaked here; see RECALL-TRACKER notes).
- **Success metric:** recall@10 full pipeline **+3pp → ≥0.21**, hit@10 **→ ≥0.74**,
  no per-stratum regression. Verify on the diagnostic harness
  (`run_diagnose.sh`), not just eval-set loss.
- **Risk to watch:** prior reranker FT runs transferred poorly across strata
  (P9: reranker hurt doc-intent −1.5pp). Keep the stratum-gated skip (FIX P10-A2)
  and re-measure per stratum.

## TARGET 2 — Embeddings (code retrieval) · SECONDARY

**Why second:** raises recall@pool itself (today's 0.477 ceiling). 60% of all
expected files never enter the top-200 — the embedding can't tell that a file
*implements* a concept it never *names* (e.g. "document preview" → `FullScreenModal.tsx`).

- **Current model:** CodeRankEmbed (code tower).
- **Training data:** `bench_runs/training_data/retrieval_pairs.jsonl` — **6391**
  `(query, expected_file)` positives. All 6391 are indexed → pure model gap,
  no indexing work needed. Hard negatives for contrastive loss = the wrong
  files currently retrieved for the same query (derivable from full_diagnose.json).
- **Success metric:** recall@pool (retrieval-only, reranker off) **+5pp → ≥0.53**.
- **Note:** embedding input should include `repo_name` + `file_path` context —
  the path carries task-level signal the file body lacks.

## NOT a training problem (do not chase with models)

- **huge_task:** ~50 queries have 16-183 expected files. recall@10 is
  structurally capped — a query with 40 expected files maxes at 10/40=0.25.
  Theoretical max recall@10 over the whole set = **0.766** even with a perfect
  pipeline. Fix on the eval side: report recall@50/@100, or split huge tickets.
- **Ground-truth quality:** BO-1439, CORE-2468, CORE-2514, CORE-2558 flagged
  (query/GT divergence, GT includes generated docs). Clean before trusting deltas.

## Operational

- Training runs on RunPod ($, ~hours). NOT auto-launched — needs explicit GO.
- Pre-train sanity: compare 5 train / 5 test positive rows before every run
  (cheap check, catches data bugs).
- Datasets are regenerated by `scripts/eval/extract_training_pairs.py
  bench_runs/diagnose/fixGH/full_diagnose.json` — rerun after any pipeline change
  so pairs reflect the live candidate distribution.

## STRUCTURAL lessons (the real rakes)
1. **A reranker cannot rank what retrieval didn't fetch.** 2026-04-18: r@25 was
   identical across ALL reranker models — the candidate pool was too narrow.
   → recall@pool (today 0.477) is the hard cap for any reranker FT. Measure it
   first; if a file isn't in the pool, reranker training is wasted on it.
2. Tiny-pair FT (10 pairs) → overfit/collapse. Need thousands (we have 3054/6391).
3. Aggressive multi-epoch FT can erase base knowledge (CoSENT −17.7pp).
4. Infra: ephemeral RunPod volumes wiped FT models mid-build → use --volume-gb=20,
   standard SSH key, poll-until-complete. 4-query eval can't discriminate → need
   ≥30 queries with multiple positives.
5. The old "96.4% recall" (2026-03) is a DIFFERENT metric (repo-level analyze_task
   + Gemini reranker, pre local-migration) — not comparable to today's file-level recall@10.

## Implication for THIS plan
- Reranker FT (Target 1) is the only model lever with a prior win (l12 +3.31pp).
  It is now better-positioned than ever: FIX-A/D gave it a cleaner, deduped
  200-pool — exactly the "pool has room" precondition the 2026-04-18 finding said
  was missing. Proceed, but gate per-stratum.
- Embedding FT (Target 2) carries a 6/6 failure prior. Treat as LOW-confidence;
  only attempt after reranker FT, with conservative settings and the regression gate.
