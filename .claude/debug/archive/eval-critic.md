# eval-critic — Phase-1 attack on doc_intent_eval_v1.jsonl

**Target:** `profiles/pay-com/doc_intent_eval_v1.jsonl` (44 queries, labeler=auto-heuristic-v1)
**Date:** 2026-04-24
**Mode:** adversarial — find real bias before $5-10 RunPod spend

---

## Executive verdict

**FIX-EVAL-BEFORE-A/B.** Two confirmed bias sources are severe enough to invalidate cross-model
recall@10 comparisons:

| Hyp | Status | Severity | Headline |
|---|---|---|---|
| H1 (representativeness) | confirmed-partial | **HIGH** | top-30 prod terms covered (28/30) but mass distribution wrong: payout=193 prod vs 4 eval (48× under-rep), provider=117 vs 5 (23×), webhook=97 vs 6 (16×) |
| H2 (labeler bias to current model) | **confirmed-severe** | **CRITICAL** | 45% of 220 expected_paths are stock auto-docs (data-layer/codebase-map/architecture/README/dev-workflow/AI-CODING-GUIDE/performance-profile/security-overview/GOTCHAS); 25% (11/44) of rows have ALL 5 paths = stock auto-docs. A model that finds better provider-specific docs gets penalized. |
| H3 (train/eval leakage) | **confirmed** | MEDIUM | 27.3% of eval rows reuse a train (repo,path) pair as a positive label; 7.3% of (row,path) instances overlap. Hardest case: `grpc-apm-payper/docs/GOTCHAS.md` appears in 4 distinct eval rows AND in train. |
| H4 (missing payper-new files) | **excluded** | none | 0/220 expected_paths are missing from `db/knowledge.db`. |
| H5 (strata coverage) | **confirmed** | MEDIUM | 9 strata cover only 48.6% of top-30 term mass. 21 top-30 terms missed (gateway, error, integration, validation, payment, account, …). 19.4% of prod queries hit only-missed terms (not represented in eval). |

`auto-heuristic-v1` is NOT a neutral judge; it is a snapshot of the existing FTS5+path-overlap
ranker. Comparing 4 candidate models on labels generated by the current ranker rewards the
candidate whose embedding distribution most closely mimics the BM25/path-overlap baseline.

**Required fixes before A/B:**
1. Hand-label (or LLM-judge) ≥ 30/44 rows where stock auto-docs dominate; replace with provider-specific or reference docs that *answer the query*.
2. Drop the (repo, path) overlap with TRAIN — re-pick alternative positives for the 12 leaked rows.
3. Up-weight `payout` / `gateway` / `validation` strata to match prod term-mass (add 10-15 prod-sampled rows).

Without these, a model winning recall@10 on this eval may *lose* on real prod traffic.

## H1 — 44 queries don't represent 1242 prod doc-intent queries

**Hypothesis:** Eval token distribution diverges materially from prod top-30; a model wins the eval by overfitting to its narrow vocabulary.

### Reproducer
```bash
cd ~/.code-rag-mcp && python3.12 -c "
import json, re
from collections import Counter
TOKEN_RE = re.compile(r'[A-Za-z][A-Za-z0-9_-]{1,}')
STOP = frozenset({'the','a','an','of','for','to','in','is','and','or','on','at','with','by','from','as','this','that','it','be'})
_DOC_QUERY_RE = re.compile(r'\b(test|tests|spec|specs|docs?|documentation|readme|guide|guides|tutorial|checklist|framework|matrix|severity|sandbox|overview|reference|rules)\b', re.I)
_CODE_SIG_RE = re.compile(r'(?:\b[a-z][a-zA-Z0-9]*\([^)]*\)|\b[A-Z][A-Z0-9_]{2,}\b|[a-z]+_[a-z_]+|\.(?:js|ts|py|go|proto)\b)')
_REPO_TOKEN_RE = re.compile(r'\b(?:grpc-|express-|next-web-|workflow-|k8s-)[a-z0-9-]+\b', re.I)

def wd(q):
    if _DOC_QUERY_RE.search(q or ''): return True
    if not q: return False
    if _CODE_SIG_RE.search(q) or _REPO_TOKEN_RE.search(q): return False
    return 2 <= len(q.split()) <= 15

def toks(s):
    return [t.lower() for t in TOKEN_RE.findall(s or '') if t.lower() not in STOP and len(t) >= 2]

prod = set()
for line in open('logs/tool_calls.jsonl'):
    d = json.loads(line)
    if d.get('tool') != 'search': continue
    q = ((d.get('args') or {}).get('query') or '').strip()
    if q and wd(q): prod.add(q)

eval_q = [json.loads(l)['query'] for l in open('profiles/pay-com/doc_intent_eval_v1.jsonl')]
pc = Counter(); ec = Counter()
for q in prod:
    for t in toks(q): pc[t] += 1
for q in eval_q:
    for t in toks(q): ec[t] += 1

print('term       prod  eval  ratio')
for t,n in pc.most_common(20):
    e = ec.get(t,0)
    print(f'{t:12s} {n:4d}  {e:3d}  {n/max(e,1):.1f}x')
"
```

### Result
| term | prod | eval | under-rep ratio |
|---|---:|---:|---:|
| payout | 193 | 4 | **48.3×** |
| nuvei | 120 | 7 | 17.1× |
| provider | 117 | 5 | 23.4× |
| webhook | 97 | 6 | 16.2× |
| method | 64 | 4 | 16.0× |
| payper | 60 | 2 | 30.0× |
| interac | 52 | 4 | 13.0× |
| error | 41 | 1 | 41.0× |
| integration | 39 | 0 | ∞ |
| paysafe | 38 | 2 | 19.0× |
| field | 33 | 0 | ∞ |
| validation | 32 | 1 | 32.0× |

28/30 top-prod terms appear in eval, but **mass is wrong by 1-2 orders of magnitude**. The
2860-search prod log has `payout` as #1 head term. Eval has it 4 times. A candidate model
that excels on `payout` queries gains nothing on this eval.

### Verdict: **confirmed-partial**
Top-term *coverage* is fine (28/30 → contradicts the 9/30 figure in the previous analysis,
which used a smaller sample of 30 eval queries). But *mass distribution* is heavily skewed
away from real prod traffic. The eval doesn't test the head of the distribution at scale.

## H2 — CRITICAL: labeler is biased toward what current ranker already finds

**Hypothesis:** `auto-heuristic-v1 = top-5 BM25 ∪ path-overlap≥0.5` produces expected_paths that
mirror the current FTS5+path-overlap ranker. Models with different inductive biases (gte-large,
nomic-v2-moe, etc.) that surface better-but-different docs get penalized.

### Reproducer
```bash
cd ~/.code-rag-mcp && python3.12 -c "
import json
STOCK = {'data-layer.md','codebase-map.md','architecture.md','README.md','dev-workflow.md','AI-CODING-GUIDE.md','performance-profile.md','security-overview.md','GOTCHAS.md','CLAUDE.md','ARCHITECTURE.md'}
rows = [json.loads(l) for l in open('profiles/pay-com/doc_intent_eval_v1.jsonl')]
total = sum(len(r['expected_paths']) for r in rows)
stock = sum(1 for r in rows for p in r['expected_paths'] if p['file_path'].rsplit('/',1)[-1] in STOCK)
all_stock_rows = sum(1 for r in rows if all(p['file_path'].rsplit('/',1)[-1] in STOCK for p in r['expected_paths']))
print(f'stock auto-doc paths: {stock}/{total} = {100*stock/total:.1f}%')
print(f'rows where ALL 5 paths are stock: {all_stock_rows}/{len(rows)} = {100*all_stock_rows/len(rows):.1f}%')
"
```

### Result
- **45.0%** of 220 expected_paths are stock auto-docs (data-layer.md, codebase-map.md, architecture.md, README.md, dev-workflow.md, AI-CODING-GUIDE.md, performance-profile.md, security-overview.md, GOTCHAS.md, CLAUDE.md, ARCHITECTURE.md).
- **25.0%** (11/44) eval rows have ALL 5 expected_paths = stock auto-docs.
- **75.0%** of rows have at least one stock auto-doc.

### Confirmation: alternative non-stock provider docs DO exist for "all-stock" rows
Sample queries where *only* stock paths were chosen, but real provider/reference docs also match:
- **prod_032** "trustly payment retry" — labeler picks 5× `grpc-apm-trustly/docs/docs/{architecture,codebase-map,dev-workflow,AI-CODING-GUIDE,performance-profile}.md`. Real BM25 in `provider_doc/reference/dictionary/flow_annotation/gotchas/domain_registry` finds `trustly-docs/.../trustly-recurring-payments.md` and `trustly-docs/.../payments.md` — content actually about retry behavior. These were rejected by the labeler because BM25 ranked stock docs higher.
- **prod_007** "gotchas global proto contract routing upstream vault-bankaccounts existing token provider" — labeler picks 5 stock paths. The query *literally asks for "gotchas"*; `global-conventions/docs/GOTCHAS.md` is ranked but mixed with `architecture.md`/`codebase-map.md`/`data-layer.md` of other repos.
- **prod_017** "Add Bulk Task Creation Method With Grouped Notifications" — looks like a Jira title, not a doc query. Labeler picks 5 stock auto-docs from 4 different workflow repos. No coherent expected answer exists for this query type.

### Why this is critical
A two-tower swap (`gte-large-en-v1.5` or `nomic-v2-moe`) will have a different cosine
landscape over auto-doc chunks vs. provider-doc chunks. The auto-docs (`data-layer.md`,
`codebase-map.md`) are short, schema-like, and semantically thin. Long-form provider docs
(`trustly-recurring-payments.md`, `nuvei-docs/.../authentication.md`) are dense English prose.
A new model may *correctly* prefer provider docs and *correctly* demote stock auto-docs —
but recall@10 against `auto-heuristic-v1` labels will *drop* because the ground truth was
built from the FTS5 ranking.

### Verdict: **confirmed — severe**
This is the dominant bias source. `auto-heuristic-v1` is not a "ground truth" — it is a
snapshot of the current ranker. The eval is structurally biased to penalize improvements.

## H3 — train/eval (repo,path) leakage

**Hypothesis:** Train v12 file (`v12_candidates_regen_labeled_FINAL.jsonl`) contains
(query, file_path) positives. If file_paths reused as eval expected_paths, models that
memorized those positives during fine-tuning get inflated recall.

### Reproducer
```bash
cd ~/.code-rag-mcp && python3.12 -c "
import json
train = set()
for l in open('profiles/pay-com/v12_candidates_regen_labeled_FINAL.jsonl'):
    d = json.loads(l)
    if d.get('query_tag') == 'doc-intent' and d.get('label_final') == '+':
        train.add((d.get('repo_name',''), d['file_path']))
ev = [json.loads(l) for l in open('profiles/pay-com/doc_intent_eval_v1.jsonl')]
pair_overlap = sum(1 for r in ev for p in r['expected_paths'] if (p['repo_name'], p['file_path']) in train)
total_pairs = sum(len(r['expected_paths']) for r in ev)
row_overlap = sum(1 for r in ev if any((p['repo_name'], p['file_path']) in train for p in r['expected_paths']))
print(f'(row,path) overlap: {pair_overlap}/{total_pairs} = {100*pair_overlap/total_pairs:.1f}%')
print(f'rows with >=1 leaked path: {row_overlap}/{len(ev)} = {100*row_overlap/len(ev):.1f}%')
"
```

### Result
- TRAIN doc-intent positives: 103 rows / 89 unique (repo, path) pairs / 63 unique file_paths
- (repo,path) overlap: **16/220 = 7.3%** of (row,path) instances
- Rows with ≥1 leaked (repo,path): **12/44 = 27.3%**
- Top leaked: `grpc-apm-payper/docs/GOTCHAS.md` (4 hits), `provider-reference-matrix/docs/references/provider-reference-matrix.md` (3), `grpc-apm-nuvei/docs/GOTCHAS.md` (2)

### Why this matters
The candidate that has been fine-tuned (or the production CodeRankEmbed which has seen these paths in v8 training) gets a small but real boost on the 12 leaked rows. It's not catastrophic (only 7.3% of (row,path) instances), but it is enough to skew a 5-pp recall comparison.

Eval should be a **disjoint** holdout. The labeler script enforces zero query overlap (line 9: "the new eval set MUST have ZERO query overlap with the doc-intent positive queries in v12_candidates_regen_labeled_FINAL.jsonl") — but it does NOT enforce path disjointness. That's the gap.

### Verdict: **confirmed**
Path disjointness should be enforced as a second-order invariant. For each leaked (repo, path) in eval, replace with the next-best non-leaked candidate from the same repo or same query strata.

## H4 — missing payper-new files

**Hypothesis:** 3 payper-new files referenced in expected_paths are absent from
`knowledge.db` (per memory `project_two_tower_v13_landed.md`: "ramo repo removed").
Recall would be artificially zero on those rows.

### Reproducer
```bash

import sqlite3, json
con = sqlite3.connect('file:db/knowledge.db?mode=ro', uri=True)
```

### Result
- **0/220 expected_paths missing**. All resolve to ≥1 chunk in `db/knowledge.db`.
- `payper-docs` repo present with `01-etransfer-payment.md`, `03-etransfer-payout.md`, `13-etransfer-status-flow.md`, `reference_interac-etransfer.md`, `reference_interac-online.md`.

### Verdict: **excluded**
The labeler script's pipeline (resolve_expected_paths queries chunks table directly)
guarantees no missing paths can be written. H4 is structurally impossible by construction.

## H5 — 9 strata miss 21 of top-30 prod terms

**Hypothesis:** Stratification picks 9 strata (payout/provider/nuvei/webhook/method/interac/refund/trustly/aircash). 21 other top-30 prod terms have no targeted coverage; long-tail queries are also undertested.

### Reproducer
```bash

import json, re
from collections import Counter
TOKEN_RE = re.compile(r'[A-Za-z][A-Za-z0-9_-]{1,}')

_REPO_TOKEN_RE = re.compile(r'\b(?:grpc-|express-|next-web-|workflow-|k8s-)[a-z0-9-]+\b', re.I)
def wd(q):
    if _DOC_QUERY_RE.search(q or ''): return True
    if not q or _CODE_SIG_RE.search(q) or _REPO_TOKEN_RE.search(q): return False
    return 2 <= len(q.split()) <= 15
def toks(s):

STRATA = {'payout','provider','nuvei','webhook','method','interac','refund','trustly','aircash'}
mass_top30 = sum(pc[t] for t in top30)
```

- 19.4% (243/1251) prod queries hit a missed-term but no strata term.

### Verdict: **confirmed — medium**
Strata bias the eval toward provider names + 1 verb. The eval does NOT test:
- "gateway error response validation" style queries (gateway=46, error=41, validation=32, response=36 prod hits)
- payper-specific (payper=60 in prod, payper-only strata gets 0 explicit slots — covered indirectly by `interac` strata picks)
- long-tail 40% (no expansion plan).

## Two additional findings

### Bonus 1 — query type mix is "search-developer pattern" not user pattern
34/44 (77%) eval rows are sampled from a 5-day window (Apr 12-21). 8 queries are bare
Jira issue titles ("Add Bulk Task Creation Method With Grouped Notifications") that have
no clear correct answer — they're search-tool-fishing-for-jira-context, not "find me docs about X".
A model that does well on prose user questions may underperform if half the eval is
copy-pasted Jira ticket headers.

### Bonus 2 — "all-stock" rows are non-evaluable, not just biased
The 11 rows where all 5 expected_paths are auto-generated docs measure NOTHING about
docs-tower quality. Both old and new models will both find or both miss those paths
based on their lexical coverage of the file boilerplate, not on doc-text semantics.
**These 11 rows should be dropped or relabeled — they're noise, not signal.**

## Concrete fix plan (1-2 hour effort, blocks A/B)

1. **Drop or relabel 11 all-stock rows** (`prod_007, 017, 020, 029, 032, 035, 039, 040, 042, 043, 044`).
   - For each, run BM25 restricted to file_type ∈ {provider_doc, reference, dictionary, flow_annotation, gotchas, domain_registry}, take top 3, accept if title/path is semantically related.
   - If no semantic match found → drop the row entirely.

2. **Eliminate 16 (repo, path) train-leakage instances.** For each leaked path, replace with the same-strata next-best non-leaked candidate. Add a post-write validator to `build_doc_intent_eval.py`:
   ```python
   train_paths = {(d['repo_name'], d['file_path']) for d in TRAIN if d['label_final']=='+'}
   for r in out_rows:
       leaked = [(p['repo_name'], p['file_path']) in train_paths for p in r['expected_paths']]
       if any(leaked): print(f'WARN: row {r["id"]} has {sum(leaked)}/5 train-leaked paths')
   ```

3. **Add 10-15 prod-sampled rows for under-represented head terms.** Specifically: payout (4 → 12), gateway (2 → 6), validation (1 → 4), error (1 → 4). Use the same pipeline but slot under "other" with explicit term filter.

4. **Re-issue with `labeler="auto-heuristic-v1+manual-fix"`**, version bump to `doc_intent_eval_v2.jsonl`, and version-pin the A/B test inputs.

If steps 1-3 cannot be completed before the A/B run, **defer the A/B**. Do not interpret
recall@10 deltas <5pp on the current eval as significant — H2 alone produces noise of that
magnitude.

## What WOULD make this eval trustworthy

A small (~30 row) **gold subset** with hand-labeled expected_paths from the user, mixed
with the 44-row auto labels, would let the synthesis stage compute *labeler agreement*
(how often does auto-heuristic-v1 match human-judged answers?). Without this, every recall
delta on doc_intent_eval_v1.jsonl is uninterpretable — we cannot tell whether candidate X
is "better" or "differently biased".
