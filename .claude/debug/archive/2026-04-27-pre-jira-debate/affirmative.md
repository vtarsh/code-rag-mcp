# AFFIRMATIVE position — split reranker into 2 (docs + code)

## Argument 1: Run 1 dual-axis evidence is a textbook negative-transfer signature, not noise
The Run 1 numbers from `.claude/debug/run1_status.md` are the smoking gun: combined-trained mxbai gains **+1pp R@10 on docs** but **loses -4.7pp on code** (0.1756 → 0.1288); combined-trained l12 does the mirror image — **+1.1pp on code, -6pp on docs** (0.25 → 0.1891). When two candidates trained on the *same* dataset oscillate symmetrically across two axes, that is not a model-capacity problem — it is the loss landscape pulling the head between two incompatible relevance distributions. The combined train.jsonl is **5,362 docs / 904 code = 5.93:1 imbalance**; gradient updates from prose-style "what is webhook" pairs literally overwrite the gradient signal from `func()`-shaped code pairs every step. Splitting the head removes the conflict by construction — each model converges on a single distribution.

## Argument 2: Routing infrastructure is already paid-for; marginal cost is one extra train
`src/search/hybrid.py:435,708,841` already calls `_query_wants_docs()` / `is_doc_intent` and `_should_skip_rerank()` — i.e., the inference path *already branches on intent* (P10 A2 ships stratum-gated rerank-skip for nuvei/aircash/trustly/webhook/refund). Adding a second reranker behind the same branch is ~30 LOC, not architecture. Training cost: docs-only subset (~5.4k pairs) and code-only subset (~0.9k pairs) are *smaller* than the 20,695-pair combined run. On rtx4090 ($0.69/hr) one cycle ≈ $1–1.5 each → **~$3 total budget hit**, well inside the $5–10 envelope. The already-rejected "one big model with twice the data" alternative (per `positions.md`) is exactly what we have today and what Run 1 just falsified.

## Argument 3: Latency budget has slack and routing keeps p95 unchanged per query
A query is *either* doc-intent or code-intent at inference — never both — so p95 is `max(docs_p95, code_p95)`, not their sum. Run 1 docs-mxbai p95 = **851ms**, code-l12 p95 = **500ms**; both individually clear the **<2s end-to-end** target with >1s headroom for retrieval + I/O. The bge-reranker-v2-m3 (568M, already-rejected for 5x cost) showed there *is* a real ceiling; mxbai-base (184M) and L-12 sit comfortably below it. Memory: only the routed model's weights need be hot in RAM per request — 16GB Mac is fine.

## Strongest counter (acknowledged): code subset is too small (904 pairs) to fine-tune a CrossEncoder without overfit
This is the Run 0 v0 wound from memory `project_runpod_stage_c_landed.md` — 10-pair MNRL FT overfit and lost -10.8pp. Real concern. **Rebuttal**: (a) 904 pairs is **90× larger** than the v0 disaster and an order of magnitude past published CrossEncoder FT thresholds (ms-marco-MiniLM transfer studies show useful signal from ~500 pairs with proper margin loss); (b) we don't have to FT from scratch — keep ms-marco-L6 *as-is for code* (Run 1 baseline L6 already wins code R@10 vs both FT candidates) and only FT the **docs head on the larger 5.4k subset**, where Run 1 mxbai-FT already shows +1pp despite the gradient conflict. That gives us a true **specialist-docs + stock-code** split, removes negative transfer, costs one train cycle ($1.5), and risks zero code regression because code path is unchanged from production.
