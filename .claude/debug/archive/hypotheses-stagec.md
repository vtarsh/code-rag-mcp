## H11: sentence-transformers 3.x model.fit() is deprecated — training may fail on pod with ST 3.x
**Axis:** env
- evidence: `scripts/runpod/setup_env.sh:21` installs `"sentence-transformers>=3.0,<4.0"`. sentence-transformers 3.0 introduced `SentenceTransformerTrainer` (HF-trainer-based) as the canonical training API; `model.fit(...)` is a legacy wrapper that still exists but is less tested. `train_docs_embedder.py:84-91` uses `model.fit(train_objectives=..., steps_per_epoch=..., epochs=1, warmup_steps=..., optimizer_params=..., show_progress_bar=True)` — older ST 2.x pattern. On ST 3.x this may hit `DeprecationWarning` at best, or silent parameter dropping at worst (warmup_steps semantics changed). Especially `losses.MultipleNegativesRankingLoss(model)` requires batch of >=2 InputExample — with `--subset=10` and `batch_size=16` in `DataLoader(examples, shuffle=True, batch_size=16)`, loader yields ONE batch of 10 samples = in-batch negatives pool of 9; marginal but not broken.
- test: `python -c "from sentence_transformers import __version__; print(__version__)"` — check local version Mac already has (tests import this module). `python -c "from sentence_transformers import SentenceTransformer; m=SentenceTransformer('nomic-ai/nomic-embed-text-v1.5', trust_remote_code=True); help(m.fit)"` — inspect signature. Also: on ST 3.x, `model.fit` internally falls back to HF Trainer with checkpoint_path defaulting to `None` — may not persist checkpoints the way ST 2.x did. Real falsification = dry-run train on CPU with subset=3 locally on Mac.
- counter-evidence: Local Mac has sentence-transformers 5.3.0. Inspected `SentenceTransformer.fit` signature directly: `(self, train_objectives, evaluator=None, epochs=1, steps_per_epoch=None, scheduler='WarmupLinear', warmup_steps=10000, optimizer_class=AdamW, optimizer_params={'lr': 2e-05}, weight_decay=0.01, ...)`. ALL params A's code passes (`train_objectives`, `steps_per_epoch`, `epochs=1`, `warmup_steps`, `optimizer_params`, `show_progress_bar`) are still accepted. Source reveals that ST 3.x+ internally translates `steps_per_epoch` → `max_steps` for HF Trainer, and the HF Trainer's `max_steps > len(dataset)/batch_size` case is handled by cycling the dataloader — not a broken path. Also `losses.MultipleNegativesRankingLoss(model)` is still the canonical in-batch-negatives loss in ST 5.3. With subset=10 and batch_size=16, DataLoader yields 1 batch of 10 examples — MNRL still works with batch ≥2, giving 9 negatives per query. Functional on both ST 3.x and 5.x.
- verdict: excluded. model.fit API is still fully functional in ST 3.0-5.x; A's "may fail" framing is speculation not backed by evidence. The real training-time concerns are H2 (HF_TOKEN order) and H9 (prefix mismatch), not the fit API. Even if ST emits DeprecationWarnings on ST 5.x, the pod installs `>=3.0,<4.0` per setup_env.sh:21, and 3.x handles this pattern cleanly.
