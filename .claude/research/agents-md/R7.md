## Executive Summary

**Recommendation:** Adopt a **Tiered Hybrid Convention** that optimizes for AI parseability first (wikilinks for inter-AGENTS.md navigation) and GitHub compatibility second (standard markdown links for code/config references).

| Target | Primary Syntax | Fallback / Companion |
|--------|---------------|---------------------|
| AGENTS.md ↔ AGENTS.md (internal catalog) | `[[relative/path.md\|Display]]` | plaintext OK; add markdown link only for "hub" references |
| AGENTS.md → code / config files | `[`src/file.py`](src/file.py)` | optional `([[src/file.py]])` suffix for LLM graph extraction |
| AGENTS.md → generated / ephemeral files | `[path](path)` or plain path in backticks | no wikilink (these files are not graph nodes) |

**Why:** `[[...]]` is instantly recognizable to LLMs as a structural reference edge, and it activates Obsidian/Foam/Logseq graph navigation for human users who open the repo in those tools. However, GitHub/GitLab render `[[...]]` as plaintext, so code-file references (where clickability matters) should remain standard markdown.

## Research Findings

### 1. How Obsidian, Foam, and Logseq handle wikilinks `[[...]]`

| Tool | Wikilink Support | Relative Path Resolution | Alias (`\|`) | Auto-backlink pane |
|------|------------------|--------------------------|--------------|-------------------|
| **Obsidian** | Native. `[[filename]]` autocompletes and creates graph edges. | Resolves by unique filename across vault; relative paths work. | Yes — `[[path\|Display]]` | Yes ("Linked mentions") |
| **Foam (VS Code)** | Native. Generates *Link Reference Definitions* at file bottom for markdown compatibility. | Resolves by filename; supports `[[dir/filename]]`. | Yes | Yes (backlinks panel) |
| **Logseq** | Native. `[[Page Name]]` creates pages in `pages/` by default; can use folder paths. | Relative paths supported; unique-name resolution. | Yes (`[[path\|alias]]`) | Yes |

**Portability issues:**
- `[[...]]` is **not** CommonMark or GFM. Standard markdown parsers (GitHub, GitLab, Pandoc, Hugo, Jekyll without plugins, VS Code default preview) render `[[...]]` as **plaintext** or broken links.
- Foam mitigates this by appending link-reference definitions (e.g., `[filename]: filename "Title"`), but these only make `[filename]` (single-bracket) work in standard parsers — `[[filename]]` itself **still does not render as a link on GitHub**.
- If the vault is ever exported to a static site, wikilinks require transpilation (e.g., `jekyll-wikilinks`, `marked-wikirefs`, or the `zettel-link-rewriter` Python script).

### 2. GitHub & GitLab rendering of `[[...]]`

- **GitHub:** Confirmed — `[[...]]` renders as plaintext. Issue `0atman/noboilerplate#42` documents this explicitly. Foam's own docs state: *"Wikilinks don't work on GitHub"*.
- **GitLab:** GitLab Flavored Markdown (GLFM) does **not** recognize `[[...]]` in repository markdown files. The only `[[...]]` syntax documented is for **GitLab Wiki pages** specifically (`[[Home]]`, `[[Title|slug]]`), which maps to wiki URLs, not repo file paths. In a `README.md` or `AGENTS.md` inside the repository, `[[...]]` is plaintext.

### 3. The "Hybrid" Approach

The hybrid approach means using **both** `[[...]]` (for Obsidian-aware tools and LLM parsing) **and** `[text](path)` (for standard renderers) in a single document. There are three practical patterns:

| Pattern | Example | Pros | Cons |
|---------|---------|------|------|
| **A. Dual-form inline** | `[[path.md\|Display]] ([Display](path.md))` | Works everywhere; explicit | Noisy; double the text |
| **B. Wikilink primary + Markdown hub** | Use `[[...]]` in body; add a "Links" section with standard markdown | Clean body; full GitHub nav | Requires maintaining two lists |
| **C. Markdown primary + Wikilink suffix** | `[Display](path.md) ([[path.md]])` | GitHub-perfect; LLM still sees wikilink | Slightly verbose |

**Recommendation for AGENTS.md:** Use **Pattern A sparingly** (only for root-to-profile "hub" links), and **Pattern C** for important code references. For the bulk of inter-AGENTS.md links, use bare wikilinks and accept GitHub plaintext, because AGENTS.md is an *AI-navigation catalog*, not human-facing documentation.

### 4. Bidirectional vs Unidirectional Backlinks

In Obsidian/Foam, a link from `A.md` to `B.md` is **unidirectional in the file text** (`A.md` contains `[[B.md]]`). The *backlink pane* makes it bidirectionally **discoverable** at runtime, but `B.md` does not gain any injected text.

In this repo:
- We **cannot** edit source code files to insert backlinks to AGENTS.md.
- Therefore, backlinks from AGENTS.md are inherently **unidirectional outward**.
- To simulate bidirectionality, each AGENTS.md should contain a **"Referenced By"** (or `## Backlinks`) section that manually lists which other AGENTS.md files link to it. This gives AI models the reverse navigation edge without modifying non-markdown files.

Example:
```markdown
## Backlinks
- Linked from: [[AGENTS.md\|Root Catalog]] (root-level overview)
- Linked from: [[profiles/my-org/AGENTS.md\|My-Org Profile]] (sibling profile)
```

### 5. Relative Path Conventions

From **root AGENTS.md** (`/Users/vaceslavtarsevskij/.code-rag-mcp/AGENTS.md`):
- ✅ `[[profiles/pay-com/AGENTS.md]]` — relative to repo root (Obsidian vault root).
- ❌ `[[../profiles/pay-com/AGENTS.md]]` — wrong; goes above repo root.

From **profile AGENTS.md** (`profiles/pay-com/AGENTS.md`):
- ✅ `[[../../AGENTS.md]]` — traverses up to repo root.
- ✅ `[[AGENTS.md]]` — works *only if* Obsidian/Foam can resolve the filename uniquely across the vault. Since there is only one `AGENTS.md` at root, this is safe in practice.
- ✅ `[[../my-org/AGENTS.md]]` — sibling profile.

**Recommendation:** Use **relative paths from the current file's location** for clarity and portability. Avoid absolute filesystem paths (`/Users/...`).

### 6. Aliases: `[[src/config.py|Configuration]]`

Aliases are supported by Obsidian, Foam, and Logseq. They improve readability when:
- The path is long or ugly.
- The display name should be a concept, not a filename.
- Linking to the same file with different contextual meanings.

**Recommendation:** Use aliases for inter-AGENTS.md links and for conceptual references. Do **not** use aliases for code file references where the actual path is important information for the AI.

## Recommended Convention (Final)

### Rule 1 — Inter-AGENTS.md links (the "graph")
Use **wikilinks** as the canonical form. Add a standard markdown fallback only for the most important hub links.

```markdown
## Profiles
- [[profiles/pay-com/AGENTS.md\|Pay-Com Profile]] ([profiles/pay-com/AGENTS.md](profiles/pay-com/AGENTS.md))
- [[profiles/example/AGENTS.md\|Example Profile]]
- [[profiles/my-org/AGENTS.md\|My-Org Profile]]
```

### Rule 2 — AGENTS.md → code / config links
Use **standard markdown** with inline code ticks for the display text. This guarantees GitHub clickability and copy-paste convenience.

```markdown
## Core Modules
- [`src/config.py`](src/config.py) — runtime configuration loader
- [`src/embedding_provider.py`](src/embedding_provider.py) — OpenAI / local embedding client
- [`tests/conftest.py`](tests/conftest.py) — pytest fixtures and mocks
```

Optionally suffix with a wikilink if the file is a significant graph node:
```markdown
- [`src/config.py`](src/config.py) `([[src/config.py]])`
```

### Rule 3 — AGENTS.md → generated / ephemeral artifacts
Do **not** use wikilinks. These are not persistent graph nodes. Use plain text or standard markdown if a URL exists.

```markdown
- `db/vectors.lance.docs/` — LanceDB vector store (gitignored, generated)
- `logs/daemon.log` — runtime logs (gitignored, rotated)
```

### Rule 4 — Aliases
Use `[[path\|Display]]` when the display text adds clarity. Prefer filename-only wikilinks when the filename is self-describing.

```markdown
- [[profiles/pay-com/docs/gotchas/clickhouse-cdc.md\|ClickHouse CDC Gotchas]]
- [[profiles/pay-com/docs/references/payment-flow.md]]
```

### Rule 5 — Paths
Always use **relative paths from the current file**.
- Root `AGENTS.md` → `profiles/pay-com/AGENTS.md`
- `profiles/pay-com/AGENTS.md` → `../../AGENTS.md` (root) or `../my-org/AGENTS.md` (sibling)

### Rule 6 — Simulated bidirectionality
Each AGENTS.md must end with a `## Backlinks` section listing inbound references from other AGENTS.md files.

```markdown
## Backlinks
- From [[AGENTS.md\|Root Catalog]] → listed under "Pay-Com Profile"
- From [[profiles/my-org/AGENTS.md\|My-Org Profile]] → listed under "Related Profiles"
```

## Tooling Notes (Optional)

No custom build tooling is *required*. However, the following lightweight checks can prevent broken links:

1. **Regex validation** (single shell command):
   ```bash
   grep -oP '\[\[\K[^\]|]+' AGENTS.md profiles/*/AGENTS.md | \
     while read f; do test -f "$f" || echo "MISSING: $f"; done
   ```
   (Note: this naive check must resolve relative paths against the source file's directory.)

2. **Pre-commit hook** (optional future work): A small Python script that parses both `[[...]]` and `[...](...)` links in `AGENTS.md` files and verifies target existence relative to the source file.

3. **Obsidian/Foam users**: Simply open the repo root as a vault. All wikilinks will resolve automatically. VS Code users can install the **Foam** extension for graph view and backlink panels.

## AI Readability Rationale

LLMs parse raw markdown. The `[[...]]` syntax provides an **unambiguous structural signal** that "this token is a reference to another file." Compared to standard markdown links `[text](path)`, wikilinks:
- Are easier to extract with simple regex (no nested parentheses ambiguity).
- Carry semantic weight even when the alias differs from the path (`[[path|Concept]]`).
- Mirror the "entity reference" patterns used in knowledge graphs, making them ideal for an AI-orientation catalog.

Standard markdown links remain superior for code files because:
- The inline code ticks (`[`file.py`](file.py)`) clearly demarcate code artifacts.
- Clickability on GitHub is valuable for human code review.
- Code files are not "knowledge nodes" in the same sense as documentation files.
